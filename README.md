# ☕ CoffeeShop Manager

Десктопное приложение для управления ассортиментом напитков и заказами кофейни.

## Описание

CoffeeShop Manager позволяет:
- Вести каталог напитков с категориями, размерами и ценами
- Создавать и отслеживать заказы
- Фильтровать, сортировать и искать данные
- Валидировать пользовательский ввод

## Технологический стек

| Компонент | Технология |
|-----------|------------|
| Язык | Python 3.10+ |
| GUI | customtkinter |
| Хранение данных | JSON-файлы |
| Архитектура | MVP (Model-View-Presenter) |

## Архитектура

Проект построен на архитектурном паттерне **MVP (Model-View-Presenter)** с соблюдением принципов **SOLID** и **ООП**.

### Схема взаимодействия
View (GUI) ←→ Presenter (логика) ←→ Repository (данные) ←→ JSON-файлы


### Слои приложения

| Слой | Пакет | Ответственность |
|------|-------|-----------------|
| **Model** | `model/` | Описание сущностей (`dataclass`), перечисления (`Enum`) |
| **View** | `view/` | Отображение данных, приём пользовательского ввода, делегирование событий презентеру |
| **Presenter** | `presenter/` | Бизнес-логика: валидация, фильтрация, сортировка, поиск, координация View и Repository |
| **Repository** | `repository/` | Чтение/запись данных в JSON-файлы |
| **Utils** | `utils/` | Вспомогательные функции (валидаторы, константы) |

### Принципы SOLID

- **S (Single Responsibility)** — каждый класс имеет одну зону ответственности. View не знает про файлы, Repository не знает про GUI.
- **O (Open/Closed)** — новый тип хранилища (например, SQLite) добавляется реализацией интерфейса `IRepository` без изменения существующего кода.
- **L (Liskov Substitution)** — все реализации интерфейсов взаимозаменяемы (`JsonDrinkRepository` можно заменить на любую другую реализацию `IDrinkRepository`).
- **I (Interface Segregation)** — интерфейсы разделены по назначению: `IDrinkView`, `IOrderView`, `IDrinkRepository`, `IOrderRepository`, `ICategoryRepository`.
- **D (Dependency Inversion)** — презентеры зависят от абстракций (интерфейсов), а не от конкретных реализаций. Все зависимости передаются через конструкторы (Dependency Injection).

### Принципы ООП

- **Инкапсуляция** — внутреннее состояние классов скрыто за приватными атрибутами (`_field`), доступ через публичные методы.
- **Наследование** — `DrinkView` наследует от `ctk.CTkFrame` и реализует `IDrinkView`; JSON-репозитории наследуют абстрактные интерфейсы.
- **Полиморфизм** — презентеры работают с интерфейсами (`IDrinkView`, `IDrinkRepository`), не зная о конкретных реализациях.
- **Абстракция** — интерфейсы (`ABC`) определяют контракты между слоями, скрывая детали реализации.

## Сущности

### Category (Категория)
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Уникальный идентификатор |
| `name` | str | Название (Кофе, Чай и т.д.) |

### Drink (Напиток)
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Уникальный идентификатор |
| `name` | str | Название напитка |
| `category_id` | UUID | Ссылка на категорию |
| `size` | Enum (S/M/L) | Размер порции |
| `price` | float | Цена в рублях |
| `available` | bool | Доступен для заказа |

### Order (Заказ)
| Поле | Тип | Описание |
|------|-----|----------|
| `id` | UUID | Уникальный идентификатор |
| `drink_ids` | list[UUID] | Список напитков |
| `created_at` | ISO 8601 | Дата создания |
| `status` | Enum | Статус (new/in_progress/ready/cancelled) |
| `total_price` | float | Итоговая стоимость |

## Функциональность

### Напитки
- ✅ Просмотр списка в таблице
- ✅ Добавление через диалоговое окно
- ✅ Редактирование через диалоговое окно

- ✅ Поиск по названию (регистронезависимый)

### Заказы

- ✅ Создание с выбором напитков
- ✅ Редактирование (состав + статус)
- ✅ Удаление (одиночное и множественное)
- ✅ Сортировка по дате, сумме, статусу
- ✅ Фильтрация по статусу
- ✅ Поиск по ID заказа
- ✅ Автоматический расчёт суммы

- ✅ Предустановленные категории при первом запуске
- ✅ Добавление новых категорий
- ✅ Защита от удаления категории с привязанными напитками

## Установка и запуск

### Требования
- Python 3.10 или выше

### Установка

```bash
# Клонирование / копирование проекта
cd coffee_shop

# Создание виртуального окружения
python -m venv venv

# Активация (Windows)
venv\Scripts\activate

# Активация (macOS/Linux)
source venv/bin/activate

# Установка зависимостей
pip install -r requirements.txt
Запуск
Bash

python main.py
При первом запуске автоматически создаётся папка data/ с JSON-файлами и предустановленными категориями.

Обработка ошибок
Некорректный пользовательский ввод → сообщение об ошибке в диалоговом окне
Отсутствующий JSON-файл → автоматическое создание
Повреждённый JSON-файл → сообщение об ошибке
Удаление категории с напитками → запрет с объяснением причины
Заказ без напитков → ошибка валидации
