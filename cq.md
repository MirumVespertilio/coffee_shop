

# Ответы на контрольные вопросы

---

## 1. Что такое архитектура программного обеспечения и зачем она нужна?

**Архитектура ПО** — это высокоуровневая структура системы, определяющая её компоненты, их ответственности и правила взаимодействия между ними.

**Зачем нужна:**

- **Разделение ответственности** — каждый компонент отвечает за свою задачу. В нашем проекте CoffeeShop Manager это чётко видно: `model/` отвечает только за описание данных, `repository/` — только за чтение/запись в JSON, `presenter/` — только за бизнес-логику, `view/` — только за GUI.
- **Поддерживаемость** — когда нужно изменить, например, способ хранения данных (перейти с JSON на SQLite), мы меняем только один слой (`repository/`), не трогая GUI и бизнес-логику.
- **Масштабируемость** — можно добавлять новые сущности (например, «Клиент»), следуя уже существующей структуре: создать модель, репозиторий, презентер и представление.
- **Тестируемость** — благодаря интерфейсам (`interfaces.py`) можно подменить реальный репозиторий на mock-объект и тестировать презентер изолированно от файловой системы.
- **Командная работа** — разработчики могут параллельно работать над разными слоями, не мешая друг другу.

## 2. В чём основная идея MV-архитектур (MVC, MVP, MVVM, MVI)?

Основная идея всех MV-архитектур — **отделение логики представления (UI) от бизнес-логики и данных**. Буква **M** всегда обозначает Model (данные и бизнес-правила), **V** — View (пользовательский интерфейс). Различие между паттернами — в третьем компоненте, который выступает посредником:

- **MVC** — Controller (контроллер)
- **MVP** — Presenter (презентер)
- **MVVM** — ViewModel (модель представления)
- **MVI** — Intent (намерение)

В нашем проекте используется **MVP**. View (`DrinkView`, `OrderView`) не содержит никакой логики — только отображает данные и передаёт пользовательские действия презентеру. Например, при нажатии кнопки «Добавить» в `DrinkView` вызывается `self._presenter.add_drink(...)`, а вся валидация, сохранение и обновление таблицы происходит в `DrinkPresenter`.

## 3. В чём принципиальные различия между MVP, MVVM и MVC?

| Аспект | MVC | MVP | MVVM |
|--------|-----|-----|------|
| **Посредник** | Controller | Presenter | ViewModel |
| **Связь View ↔ Model** | View может напрямую читать Model | View не знает о Model | View привязан к ViewModel через data binding |
| **Роль View** | Отображает данные, может содержать логику отображения | Максимально «тупой», только отображает и передаёт события | Привязывается к свойствам ViewModel |
| **Связь View ↔ Посредник** | View знает о Controller, Controller знает о View | Двусторонняя через интерфейс: Presenter держит ссылку на IView | Односторонняя через биндинг данных |
| **Тестируемость** | Средняя | Высокая | Высокая |

**В нашем проекте (MVP):**
- `DrinkView` реализует интерфейс `IDrinkView` и не знает ничего о модели `Drink` — получает только `list[dict]`
- `DrinkPresenter` держит ссылки на `IDrinkView` и `IDrinkRepository`, координирует их
- View вызывает методы презентера (`self._presenter.add_drink(...)`), а презентер вызывает методы View (`self._view.display_drinks(...)`, `self._view.show_error(...)`)

## 4. Какие задачи решает слой представления (Presentation layer)?

Слой представления отвечает за:

1. **Отображение данных пользователю** — в нашем проекте `DrinkView.display_drinks()` получает список словарей и рендерит строки таблицы с чекбоксами, названиями, ценами и т.д.

2. **Приём пользовательского ввода** — поле поиска (`_search_var`), фильтры (`_filter_category_var`, `_filter_size_var`), чекбоксы выбора, кнопки действий.

3. **Делегирование действий презентеру** — View не обрабатывает логику сам, а передаёт данные:
   ```python
   def _on_add_click(self) -> None:
       DrinkDialog(parent=..., on_save=self._presenter.add_drink)
   ```

4. **Отображение обратной связи** — сообщения об ошибках (`show_error`), успешных операциях (`show_success`), запросы подтверждения (`ask_confirmation`).

5. **Визуальное оформление** — в нашем проекте используются цветовые индикаторы: зелёная галочка для доступных напитков, красный крестик для недоступных, цветные статусы заказов (синий — новый, оранжевый — готовится, зелёный — готов, красный — отменён).

## 5. Что относится к модели (Model) в MV-архитектурах?

Model — это **всё, что связано с данными и бизнес-правилами**, не зависящими от способа отображения.

В нашем проекте к Model относятся:

1. **Сущности данных** (пакет `model/`):
   - `Category` — категория напитка с полями `id`, `name`
   - `Drink` — напиток с полями `id`, `name`, `category_id`, `size`, `price`, `available`
   - `Order` — заказ с полями `id`, `drink_ids`, `created_at`, `status`, `total_price`

2. **Перечисления**:
   - `DrinkSize` (S, M, L) — допустимые размеры порций
   - `OrderStatus` (new, in_progress, ready, cancelled) — допустимые статусы заказов

3. **Методы сериализации** — `to_dict()` и `from_dict()` для преобразования объектов в JSON и обратно.

4. **Слой доступа к данным** (пакет `repository/`) — репозитории, обеспечивающие CRUD-операции над моделями. Хотя репозитории иногда выделяют в отдельный слой, концептуально они относятся к «модели» в MV-архитектуре.

5. **Правила валидации** (`utils/validators.py`) — бизнес-правила: цена > 0, название непустое, минимум 1 напиток в заказе и т.д.

## 6. Почему пользовательский интерфейс не должен содержать бизнес-логику?

**Причины:**

1. **Тестируемость** — бизнес-логику в презентере можно тестировать без запуска GUI. Если бы валидация цены была в `DrinkDialog`, для её тестирования пришлось бы создавать окно customtkinter.

2. **Переиспользуемость** — если завтра понадобится веб-интерфейс или мобильное приложение, бизнес-логика в `DrinkPresenter` останется без изменений — нужно только создать новую реализацию `IDrinkView`.

3. **Поддерживаемость** — в нашем проекте `DrinkView` содержит ~200 строк чисто визуального кода. Если бы туда добавить валидацию, фильтрацию, сортировку, работу с файлами — файл бы вырос до 500+ строк и стал нечитаемым.

4. **Принцип единственной ответственности (SRP)** — View отвечает за отображение, Presenter — за логику. В нашем проекте `DrinkView._on_add_click()` не проверяет данные, а просто передаёт callback `self._presenter.add_drink` в диалог.

**Пример из проекта:** Фильтрация напитков. `DrinkView` только хранит значение фильтра и предоставляет метод `get_filter_category()`. Сама фильтрация выполняется в `DrinkPresenter._apply_filters()`:
```python
cat_filter = self._view.get_filter_category()
if cat_filter:
    drinks = [d for d in drinks if d.category_id == cat_filter]
```

## 7. Что такое принципы SOLID и для чего они применяются?

**SOLID** — пять принципов объектно-ориентированного проектирования, которые помогают создавать гибкий, поддерживаемый и расширяемый код.

### S — Single Responsibility Principle (Принцип единственной ответственности)
Каждый класс должен иметь одну причину для изменения.

**В проекте:** `JsonDrinkRepository` отвечает только за чтение/запись напитков в JSON. Если изменится формат хранения — меняем только этот класс. `DrinkPresenter` отвечает только за бизнес-логику напитков. `DrinkView` — только за отображение.

### O — Open/Closed Principle (Принцип открытости/закрытости)
Классы открыты для расширения, но закрыты для модификации.

**В проекте:** Если нужно хранить данные в SQLite, мы создадим `SqliteDrinkRepository`, реализующий `IDrinkRepository`, не изменяя ни `DrinkPresenter`, ни `DrinkView`. Презентер работает с интерфейсом, а не с конкретной реализацией.

### L — Liskov Substitution Principle (Принцип подстановки Лисков)
Объекты подклассов должны быть взаимозаменяемы с объектами базового класса.

**В проекте:** `JsonDrinkRepository` можно заменить на любую другую реализацию `IDrinkRepository`, и `DrinkPresenter` продолжит работать корректно, потому что он зависит от интерфейса.

### I — Interface Segregation Principle (Принцип разделения интерфейсов)
Клиенты не должны зависеть от методов, которые они не используют.

**В проекте:** Интерфейсы разделены: `IDrinkRepository`, `IOrderRepository`, `ICategoryRepository` — каждый содержит только методы, необходимые для работы с конкретной сущностью. Аналогично `IDrinkView` и `IOrderView` — отдельные интерфейсы с разными методами.

### D — Dependency Inversion Principle (Принцип инверсии зависимостей)
Модули верхнего уровня не должны зависеть от модулей нижнего уровня. Оба должны зависеть от абстракций.

**В проекте:** `DrinkPresenter` в конструкторе принимает `IDrinkView` и `IDrinkRepository` (абстракции), а не `DrinkView` и `JsonDrinkRepository` (конкретные классы). Сборка зависимостей происходит в `main.py`:
```python
DrinkPresenter(view=drink_view, drink_repo=drink_repo, category_repo=category_repo)
```

## 8. Что такое абстракция в объектно-ориентированном программировании?

**Абстракция** — это выделение существенных характеристик объекта, скрывая несущественные детали. Позволяет работать с объектами на уровне «что они делают», а не «как они это делают».

**В нашем проекте абстракция реализована через:**

1. **Абстрактные базовые классы (ABC):**
   - `IDrinkRepository` определяет контракт: «должны быть методы `get_all()`, `add()`, `update()`, `delete()`». Как именно данные хранятся (JSON, SQLite, API) — неважно для потребителя.
   ```python
   class IDrinkRepository(ABC):
       @abstractmethod
       def get_all(self) -> list[Drink]: ...
       @abstractmethod
       def add(self, drink: Drink) -> None: ...
   ```

2. **Интерфейсы View:**
   - `IDrinkView` абстрагирует представление: «должен уметь отображать напитки, показывать ошибки, возвращать выбранные ID». Конкретная реализация может быть на customtkinter, PyQt, или даже в консоли.

3. **Dataclass-ы как абстракции данных:**
   - `Drink` абстрагирует понятие «напиток» — представляет его как набор атрибутов, скрывая детали хранения (JSON-поля, типы данных файла).

---

## 9. Чем интерфейс отличается от абстрактного класса?

| Аспект | Интерфейс | Абстрактный класс |
|--------|-----------|-------------------|
| **Назначение** | Определяет контракт (что должен уметь объект) | Определяет контракт + частичную реализацию |
| **Реализация методов** | Только сигнатуры, без реализации | Может содержать реализованные методы |
| **Множественное наследование** | Класс может реализовать несколько интерфейсов | Обычно наследуется один абстрактный класс |
| **Состояние** | Не содержит полей/атрибутов | Может содержать атрибуты |

**В Python** формального разделения нет — и то, и другое реализуется через `ABC`. Но по смыслу различие сохраняется.

**В нашем проекте:**

- `IDrinkView` и `IDrinkRepository` — по сути **интерфейсы**: содержат только абстрактные методы без реализации, определяют контракт.
  ```python
  class IDrinkRepository(ABC):
      @abstractmethod
      def get_all(self) -> list[Drink]: ...
  ```

- `DrinkView` реализует **два интерфейса** одновременно (множественное наследование): наследует от `ctk.CTkFrame` (визуальный компонент) и `IDrinkView` (контракт представления):
  ```python
  class DrinkView(ctk.CTkFrame, IDrinkView):
  ```

Если бы у нас было общее поведение для всех репозиториев (например, базовая логика кэширования), мы бы сделали **абстрактный класс** `BaseRepository` с частичной реализацией и наследовали бы от него.

## 10. Что такое полиморфизм и какие его виды существуют?

**Полиморфизм** — возможность использовать объекты разных типов через единый интерфейс. «Один интерфейс — множество реализаций».

### Виды полиморфизма:

**1. Полиморфизм подтипов (наследования):**
Объекты разных классов, реализующих один интерфейс, используются взаимозаменяемо.

В нашем проекте `DrinkPresenter` работает с `IDrinkRepository`:
```python
def __init__(self, view: IDrinkView, drink_repo: IDrinkRepository, ...):
```
Ему можно передать `JsonDrinkRepository`, `SqliteDrinkRepository` или любую другую реализацию — код презентера не изменится.

**2. Параметрический полиморфизм (обобщения/generics):**
Код работает с любым типом данных. В Python это `list[Drink]`, `dict[str, str]` — коллекции, параметризованные типом.

В проекте: `list[Category]`, `list[Drink]`, `list[Order]` — репозитории возвращают типизированные списки.

**3. Ad-hoc полиморфизм (перегрузка):**
Одна функция ведёт себя по-разному в зависимости от типа аргументов. В Python реализуется через проверку типов или `@singledispatch`.

В проекте: метод `to_dict()` есть у `Category`, `Drink` и `Order` — каждый сериализует себя по-своему, но интерфейс одинаковый.

**4. Утиная типизация (duck typing):**
Если объект имеет нужные методы — он подходит, независимо от формального наследования. Характерно для Python.

В проекте: `CategoryDialog` обращается к `self._view_ref` через `hasattr(self._view_ref, "_presenter")` — не проверяет тип, а проверяет наличие атрибута.

## 11. Что такое слой доступа к данным и зачем он выделяется отдельно?

**Слой доступа к данным (Data Access Layer, DAL)** — компонент, инкапсулирующий всю логику работы с источником данных: чтение, запись, обновление, удаление.

**В нашем проекте** это пакет `repository/`:
- `JsonCategoryRepository` — чтение/запись категорий в `categories.json`
- `JsonDrinkRepository` — чтение/запись напитков в `drinks.json`
- `JsonOrderRepository` — чтение/запись заказов в `orders.json`

**Зачем выделять отдельно:**

1. **Независимость от источника данных** — презентер вызывает `self._drink_repo.get_all()` и не знает, что данные хранятся в JSON. Если завтра перейдём на PostgreSQL, создадим `PostgresDrinkRepository` и подставим в `main.py` — остальной код не изменится.

2. **Единая точка доступа** — вся работа с файлами сосредоточена в одном месте. Не нужно искать `json.load()` по всему проекту.

3. **Обработка ошибок** — ошибки ввода-вывода обрабатываются в одном слое:
   ```python
   except (json.JSONDecodeError, KeyError) as e:
       raise IOError(f"Ошибка чтения файла напитков: {e}")
   ```

4. **Автоматическая инициализация** — `_ensure_file_exists()` создаёт файлы и дефолтные категории при первом запуске. Эта логика не должна быть ни в GUI, ни в бизнес-логике.

## 12. Какие преимущества даёт слоистая архитектура по сравнению с монолитной?

**Монолитная архитектура** — весь код в одном файле или модуле без чёткого разделения. Типичная ситуация: обработчик кнопки читает JSON, валидирует данные, обновляет таблицу.

**Слоистая архитектура** нашего проекта имеет 4 слоя: Model → Repository → Presenter → View.

| Преимущество | Монолит | Слоистая (наш проект) |
|---|---|---|
| **Поддерживаемость** | Изменение одной функции может сломать всё | Изменения в `repository/` не влияют на `view/` |
| **Тестируемость** | Невозможно тестировать логику без GUI | `DrinkPresenter` тестируется с mock-объектами |
| **Переиспользуемость** | Код нельзя использовать повторно | `DrinkPresenter` работает с любым `IDrinkView` |
| **Командная работа** | Конфликты при одновременном редактировании | Разработчики работают в разных пакетах |
| **Расширяемость** | Добавление функции требует изменений «везде» | Добавление SQLite = новый класс в `repository/` |
| **Читаемость** | 1000+ строк в одном файле | Файлы по 100-200 строк с чёткой ответственностью |

**Конкретный пример из проекта:** Если нужно добавить экспорт в CSV, мы добавим метод в `repository/` или создадим новый модуль `utils/export.py`, не трогая GUI и бизнес-логику.

## 13. Какие основные слои присутствуют в архитектуре приложения и за что каждый из них отвечает?

В нашем проекте CoffeeShop Manager выделены следующие слои:

### 1. Model (Модель) — `model/`
**Ответственность:** Описание структуры данных и бизнес-правил.
- `Category`, `Drink`, `Order` — dataclass-ы с полями
- `DrinkSize`, `OrderStatus` — перечисления допустимых значений
- `to_dict()` / `from_dict()` — сериализация/десериализация

### 2. Repository (Репозиторий) — `repository/`
**Ответственность:** Доступ к внешнему хранилищу данных.
- `interfaces.py` — абстрактные контракты (`ICategoryRepository`, `IDrinkRepository`, `IOrderRepository`)
- `json_*_repository.py` — конкретные реализации для JSON-файлов
- Методы: `get_all()`, `get_by_id()`, `add()`, `update()`, `delete()`, `delete_many()`

### 3. Presenter (Презентер) — `presenter/`
**Ответственность:** Бизнес-логика, координация View и Repository.
- `DrinkPresenter` — CRUD напитков, валидация, фильтрация, сортировка, поиск
- `OrderPresenter` — CRUD заказов, расчёт суммы, управление статусами
- Вызывает методы View для отображения результатов

### 4. View (Представление) — `view/`
**Ответственность:** Пользовательский интерфейс.
- `MainWindow` — главное окно с вкладками
- `DrinkView` — таблица напитков, фильтры, кнопки
- `OrderView` — таблица заказов, фильтры, кнопки
- `dialogs.py` — модальные окна создания/редактирования

### 5. Utils (Утилиты) — `utils/`
**Ответственность:** Вспомогательные функции, не привязанные к конкретному слою.
- `validators.py` — функции валидации полей
- `constants.py` — константы (пути к файлам, размеры окна, тема)

## 14. Как организовано взаимодействие между слоями приложения?

Взаимодействие организовано **строго через интерфейсы** с **однонаправленными зависимостями**:

```
View ←→ Presenter ←→ Repository ←→ JSON-файлы
         ↓
       Model
```

### Сценарий: пользователь добавляет напиток

1. **View → Presenter:** Пользователь заполняет форму в `DrinkDialog` и нажимает «Сохранить». Диалог вызывает callback `self._on_save(name, category_id, size, price, available)`, который указывает на `DrinkPresenter.add_drink()`.

2. **Presenter (валидация):** `DrinkPresenter.add_drink()` вызывает `_validate_drink_fields()`, который использует функции из `validators.py`. Если есть ошибки — вызывает `self._view.show_error(...)` и возвращает `False`.

3. **Presenter → Repository:** Если валидация прошла — создаёт объект `Drink` и вызывает `self._drink_repo.add(drink)`.

4. **Repository → JSON:** `JsonDrinkRepository.add()` читает текущий файл, добавляет новый объект, перезаписывает файл.

5. **Presenter → View (обновление):** Презентер вызывает `self.load_drinks()`, который читает данные через репозиторий, применяет фильтры/сортировку, преобразует в `list[dict]` и вызывает `self._view.display_drinks(drink_dicts)`.

6. **View (отображение):** `DrinkView.display_drinks()` очищает таблицу и рендерит новые строки.

### Ключевые правила:
- **View не знает о Repository** — не импортирует его, не вызывает напрямую
- **Repository не знает о View** — не импортирует его
- **Presenter знает оба** — но только через **интерфейсы** (`IDrinkView`, `IDrinkRepository`)
- **Сборка зависимостей** — происходит в одном месте: `main.py`

## 15. Как осуществляется обработка ошибок при работе с данными?

В проекте реализована **многоуровневая обработка ошибок:**

### Уровень 1: Валидация пользовательского ввода (Presenter)
Перед любой операцией записи данные проверяются:
```python
def _validate_drink_fields(self, name, category_id, size, price_str):
    errors = []
    err = validate_drink_name(name)  # непустое, ≤100 символов
    if err:
        errors.append(err)
    err = validate_drink_price(price_str)  # число, >0, ≤99999.99
    if err:
        errors.append(err)
    return errors
```
Если есть ошибки — они отображаются пользователю через `self._view.show_error("\n".join(errors))`, операция прерывается.

### Уровень 2: Бизнес-логика (Presenter)
Проверки, связанные с бизнес-правилами:
- Нельзя удалить категорию, если есть напитки с ней: `if linked: self._view.show_error(...)`
- Нельзя добавить в заказ недоступный напиток: `if not drink.available: self._view.show_error(...)`
- Минимум 1 напиток в заказе: `validate_order_drinks(drink_ids)`

### Уровень 3: Ошибки ввода-вывода (Repository)
Ошибки чтения/записи JSON-файлов перехватываются и преобразуются в `IOError`:
```python
def _load_all(self) -> list[Drink]:
    try:
        with open(self._file_path, "r", encoding="utf-8") as f:
            data = json.load(f)
        return [Drink.from_dict(item) for item in data]
    except (json.JSONDecodeError, KeyError) as e:
        raise IOError(f"Ошибка чтения файла напитков: {e}")
```

### Уровень 4: Обработка в Presenter
Presenter перехватывает `IOError` из Repository и показывает пользователю:
```python
try:
    self._drink_repo.add(drink)
except IOError as e:
    self._view.show_error(str(e))
```

### Уровень 5: Автовосстановление (Repository)
При отсутствии файлов — автоматическое создание:
```python
def _ensure_file_exists(self) -> None:
    os.makedirs(DATA_DIR, exist_ok=True)
    if not os.path.exists(self._file_path):
        self._save_all([])  # создаём пустой файл
```
Для категорий — создание с предустановленными значениями (Кофе, Чай, Лимонад и т.д.).

### Уровень 6: Отображение пользователю (View)
Все ошибки показываются через стандартные диалоговые окна `messagebox`:
```python
def show_error(self, message: str) -> None:
    messagebox.showerror("Ошибка", message)
```
Пользователь всегда получает понятное сообщение, а не stack trace в консоли.
